# STYLE：体例与排版规范

> 本文件规定**每一章该长什么样**。写作时请按"硬模板 → 软语气 → 排版细则 → 双版本排版"四层对照。
>
> 署名：**作者 Austin**（本书所有版本的 metadata `author` 字段均填 `Austin`）。
> 本书同时交付 **PDF** 与 **ePub** 两个版本，源稿为同一套 Markdown，构建时分流（见 §11、§12）。

---

## 1. 章节硬模板（缺一不可）

每章正文**必须**按以下顺序出现这五块：

```
# 第 N 章 章节标题

> 一句话交付：读完本章，你能多做 ____。    ← 必须单独成段、加引用块

## 章前小故事或现场片段（~150 字，可选但推荐）

## 正文（多个 ## 小节，概念与实例交替）

## 本章小结（~200 字内，3~5 条 bullet）

## 行动清单（3~5 条，每条都必须是"今天就能做"的动作）
- [ ] 动作 1（主语是"你"，动词开头，可验证）
- [ ] 动作 2
- [ ] ...
```

**硬规则**：

1. 「一句话交付」里**禁止使用"了解/理解/认识"**等虚词，必须是"你能多做什么"。
   - ❌ 了解 Prompt 工程的演进。
   - ✅ 你能把一条模糊需求改写成带角色、输入、约束、输出格式的 4 段式 Prompt，并在 5 分钟内产出首版可运行代码。
2. 「行动清单」的每一条都要能回答"**我今天下班前能做完这条吗？**"。
3. 「本章小结」**禁止重复章节标题**当作总结，要提炼**可迁移的 3~5 句话**。

## 2. 正文节奏

- **小标题多、段落短**：一个 `##` 小节控制在 600~900 字；一个段落控制在 3~6 行。
- **概念 → 实例 → 提炼**三步走：
  - 抛概念（1 段）→ 给 mini-library 的具体示例（1 段 + 代码）→ 提炼 1 句可迁移结论。
- **不要铺陈名词**：同一段里出现 3 个以上陌生术语时，拆分；或把定义挪到 `GLOSSARY.md` 引用。

## 3. 「前后对照」格式（全书 ≥ 10 组）

每当出现"坏做法 vs 好做法"的对照，统一用下面的块：

```markdown
### 对照 · [一句话主题]

**坏做法（2022 气质）**

<![CDATA[
这里给"糟糕 Prompt / 糟糕代码 / 糟糕流程"的完整样本
]]>

> 为什么坏：1~2 句话说清代价（多花 token / 多返工 / 合规风险）。

**好做法（当前版本）**

<![CDATA[
这里给修正后的完整样本
]]>

> 为什么好：1~2 句话说清收益（可迁移 / 可复测 / 可授权给模型跑）。
```

> 注：实际写作里把 `<![CDATA[...]]>` 换成对应语言的 fenced code block（```prompt / ```java / ```bash 等）。此处只是占位，避免在 STYLE.md 里 引入大量代码导致审阅噪声。

## 4. Prompt / Context / Harness 三种示例格式

为了让读者一眼看出**这是哪一代杠杆的例子**，统一用语言标签：

- Prompt 示例：使用 ```text 或 ```prompt，并在上方一行写 `示例 · Prompt #07 · UC-01 借书接口`。
- Context 包裹示例：用 ```xml 或 ```markdown 表示最终送入模型的完整 payload，结构见 `GLOSSARY.md` 的"上下文包裹顺序"条目。
- Harness 流程示例：用**伪代码 / Mermaid / Shell 管道**三选一，避免贴真实框架代码造成版本锁定。

## 5. 代码块风格

- Java 片段必须带包名（可省略 `import`），类上加最小必要注解；
- 所有代码**符合 `CASESPEC.md` 第 5 节 8 条硬约束**，除非是 `BAD-xx` 对照；
- 超过 40 行的代码块要拆分 + 解释；**禁止"大块代码不讲解"**；
- 关键行用注释标注 `// ① 入参校验` / `// ② 扣减库存` 等步骤号，正文回扣时用 `①②③`。

## 6. 语气与人称

- 用「你」，不用「我们」或「大家」；
- **不神化模型**：禁止"AI 无所不能 / 彻底改变 / 颠覆"之类词；用"当前这代模型擅长 X，但在 Y 上仍要人兜底"。
- **承认工具会变**：每次点名具体模型或产品时，加一句"截至 2026 年初"，并给可迁移的判断维度。
- 允许个人化叙述（"我在去年做 mini-library 的时候……"），但要节制，每章 ≤ 2 处。
- **禁止**空洞的 AI 未来学段落；**禁止**凑字数的排比句。

## 7. 中文排版细则

- 中英混排时**中英之间加一个半角空格**：`Prompt engineering 的第一步是……`。
- 数字与汉字之间加半角空格：`大约 30 天`、`命中率 92%`。
- 标点用**中文全角**，但代码内部标点原样保留。
- **破折号**用「——」而非「--」或「–」。
- **省略号**用「……」而非「...」。

### 7.1 核心术语的中英写法（重要）

针对 `Prompt` / `Context` / `Harness` 以及它们的组合词（`Prompt Engineering`、`Context Window`、`Harness`、`Agent`、`RAG` 等）：

- **首次出现**：英文 + 中文括注，形如 `Prompt Engineering（提示词工程）`；
- **后续所有出现**：**保留英文原词**，不再用中文替代。正确：`这一代 Prompt 的天花板`；错误：`这一代提示词的天花板`。
- 当需要用中文叙述其"动作"时，动词部分可以中文化，术语部分仍用英文。例：`把 Context 组织好` ✅；`把上下文组织好` ❌（在本书上下文里）。
- 例外：面向**非技术读者的 00 前言章**可以适度中文化（如"提示词 / 上下文 / 工作台"），但同段内必须给出英文对照；一旦进入 01 章及以后，全部英文。

## 8. 章内引用互链

- 引用其他章节：`见 03 章〈Prompt 骨架〉`，不要只写"前面说过"；
- 引用 meta 文件：`见 meta/CASESPEC.md §5 第 4 条`；
- 引用 appendix：`见附录 A〈检查清单〉`。

## 9. 字数口径

- 以 `dist/全书.md` 的 `wc -m` 汉字数为口径；
- 目标区间 **4~7 万字正文**（不含目录/脚注/代码块里的英文 token）；
- 每章字数预算见 `OUTLINE.md`。

## 10. 章级自检清单（收尾前对照）

- [ ] 有「一句话交付」，且是"你能多做什么"而不是"你能了解什么"。
- [ ] 至少 1 组「前后对照」或「坏例子 → 好例子」。
- [ ] mini-library 代码不违反 `CASESPEC.md` §5（现为 6 条硬约束）。
- [ ] 有「本章小结」+ 3~5 条「行动清单」。
- [ ] 本章没有"新宇宙"例子，或所有平行例子都标注了。
- [ ] 01 章及以后：`Prompt / Context / Harness` 均保留英文，未被中文替换。
- [ ] `wc -m` 字数落在 `OUTLINE.md` 给本章的预算 ±15% 内。

## 11. PDF 版本排版规范（对齐"花叔橙皮书"阅读节奏）

> "花叔橙皮书"在社群流传中体现出的**共识节奏**：小段落、密集代码块、清晰的引用块、醒目的小标题层级、适合平板/打印阅读。本书 PDF 版本对齐这一阅读节奏，**不逐页模仿版式**（遵守 `DELEGATION_BRIEF` 的"禁止项"第 1 条）。

### 11.1 版式硬约束

- **页面尺寸**：A4（210 × 297 mm），上下边距 22 mm，左右边距 24 mm；这是为了让一行中文正好 ~36 字，接近纸书。
- **正文字号**：11 pt / 行距 1.6；小标题 `h2 = 16 pt`、`h3 = 13 pt`；代码块 10 pt 等宽字体。
- **字体选型**（首选 → 回退）：
  - 中文：`PingFang SC` → `Noto Sans CJK SC` → `宋体`；
  - 西文：`Inter` → `-apple-system` → `system-ui`；
  - 等宽：`JetBrains Mono` → `Menlo` → `monospace`。
- **主色调**：单一强调色（橙色家族，建议 `#E8772E`），仅用于 h2 下划线、引用块左边条、代码块边框。不要大面积染色。
- **引用块（blockquote）**：左侧 4 px 橙色条，浅灰底（`#FAF6F2`），圆角 4 px；「一句话交付」一定用引用块包住。
- **代码块**：浅底深字（避免黑底白字在 A4 打印发灰），带 4 px 左边条，语言标签右上角小字灰色。
- **表格**：表头浅橙底（`#FDF1E6`），非全框线，只保留行分割线。
- **页眉页脚**：页眉左为章节编号+标题，页脚右为页码，页脚左为"《……》· Austin · 2026"。
- **TOC**：自动生成到 h3，最多 3 级。
- **封面**：橙色主色块 + 书名 + 副标题 + 作者 "Austin" + 版本号 + 构建日期。

### 11.2 对 Markdown 作者的影响

- **图片**必须有 alt 文本，且宽度用百分比（便于 PDF 适配）；
- **长表格**（> 6 列）要拆，不要横滚；
- **不要用 HTML 原生标签** 去做复杂排版，统一走 Markdown（PDF 与 ePub 共用源码）。

## 12. ePub 版本排版规范（对齐微信读书）

> 微信读书对 ePub 的约束在"同类上架书"中可总结为下列要点，本书 **ePub 版本严格对齐**这些要点，避免上架返修。

### 12.1 结构约束

- **EPUB 3.0** 规范，`nav.xhtml` 做目录（同时保留 `toc.ncx` 向后兼容）；
- **每章一个 `xhtml` 文件**（与 `chapters/NN-*.md` 一一对应）；
- `content.opf` 的 metadata 必填：
  - `dc:title` = 本书书名
  - `dc:creator opf:role="aut"` = **Austin**
  - `dc:language` = `zh-CN`
  - `dc:identifier` = UUID
  - `dc:publisher` / `dc:date` / `dc:description` 按实际填
- **封面**：单独一页 `cover.xhtml` + `cover.jpg`（建议 1200 × 1600 px，色彩空间 sRGB，文件 ≤ 500 KB）。

### 12.2 排版硬约束（避免被微信读书重排器破坏）

- **禁止固定像素字号**：不要写 `font-size: 14px`；用 `em` / `rem` / 或干脆不写（让阅读器接管）。
- **禁止自定义字体嵌入**：微信读书使用自有字体，嵌入字体会增加包体积且无效。
- **禁止固定颜色背景**（正文）：用户会切换日 / 夜 / 护眼三模式，任何 `background-color` 都会破坏夜间模式。
  - 例外：代码块可以给**柔和的浅灰底**（`#F5F5F5`），因为三模式下这个颜色仍可读。
- **段首不缩进、段间留白**：ePub 阅读场景下，段间空一行比段首缩进更清晰。
- **图片**：统一放 `images/`，宽度 `max-width: 100%`；不超过 800 px 宽；避免横向超宽图表。
- **代码块**：不用 `<pre>` 外再套 `<div>`；保证 `<pre>` 在小屏可横向滑动（微信读书会处理）；**单行不超过 80 字符**（超过会触发自动折行，丢失缩进）。
- **表格**：≤ 4 列；超过就改为**并列两段 bullet**，因为手机阅读表格会被压缩成马赛克。
- **脚注**：ePub 用 `<aside epub:type="footnote">` 而非 HTML 脚注链接，微信读书会渲染为弹窗。
- **锚点链接（章节互链）**：全部用相对路径 + `#id`，禁止绝对 URL。

### 12.3 构建产物校验点

`scripts/build_book.py` 产出 `dist/全书.epub` 后，应跑下列校验（任意一条失败都要修）：

1. `epubcheck`（如已安装）报 0 错误；
2. 书名 / 作者 / 语言字段正确；
3. 封面图在目录第一项能预览；
4. 任取 3 章，在手机微信读书"上传我的书"后：
   - 夜间模式下代码块仍可读；
   - 「一句话交付」的引用块样式保留；
   - 目录可跳转、章内锚点可跳转。

### 12.4 作者对源稿的要求

作为 Markdown 作者，你**唯一需要额外做的事**是：

- 写代码块时主动把长行拆到 80 字符内；
- 超过 4 列的表格主动拆；
- 不要在 Markdown 里塞 HTML `style=""`；
- 图片放到 `images/` 并用相对路径。

其余差异由 `scripts/build_book.py` 在构建时通过两套 CSS（`assets/print.css` / `assets/epub.css`）与模板分流处理。
