# GLOSSARY：术语表（全书统一口径）

> 本表规定全书**每个术语只有一种写法 + 一种定义**。
> 核心术语（`Prompt` / `Context` / `Harness` 及其派生）遵循以下约定：
>
> - **首次出现**（通常在 00 前言 / 01 地图章）：英文 + 中文括注，形如 `Prompt Engineering（提示词工程）`；
> - **后续所有章节**：**保留英文原词**，不做中文替代。
>
> 如需更长解释，见附录 C〈术语扩展〉。

---

## A. 三代杠杆（核心词，全书英文写法）

| 主写法 | 首次中文括注 | 定义（≤ 60 字） |
| --- | --- | --- |
| **Prompt Engineering** | 提示词工程 | 通过改写送给模型的**单轮或多轮文本**，尽量提升输出质量的做法。杠杆在"话术"。 |
| **Context Engineering** | 上下文工程 | 通过挑选、组织、压缩**喂给模型的信息包**（规则 + 数据 + 示例 + 代码）来提升输出的做法。杠杆在"喂料"。 |
| **Harness Engineering** | 框架工程 / 工作台工程 | 围绕模型搭建**角色 + 工具 + 记忆 + 评测 + 护栏**五件套，把模型当运行时的一部分来工程化的做法。杠杆在"工位"。 |

派生词全部保留英文：`Prompt` / `Context` / `Harness` / `Context Window` / `Context Budget` / `Agent` / `Tool Use` / `Few-shot` / `System Prompt` / `Guardrail`。

> 本书**不使用** "提示词工程" / "上下文工程" / "框架工程" 这种中文形式作为正文术语（除首次括注外）；
> 也**不使用** `Agent Engineering` 或 "智能体工程" 替代 `Harness Engineering`——因为 Agent 是产物，Harness 是方法。

## B. Context 包裹顺序（固定 6 段）

当本书出现「Context 包裹」，指下面这个**固定顺序**的 6 段结构：

1. **系统规则（System）**：不变的行业 / 公司 / 模型设定。
2. **项目约束（Project）**：`CASESPEC §5` 这 8 条。
3. **任务描述（Task）**：本次要做的事，含验收标准。
4. **示例（Shots）**：零到三个 few-shot。
5. **代码上下文（Code）**：被改的函数、调用链、相邻测试。
6. **输出要求（Output）**：格式、字段、长度、禁止项。

任何章节讨论 context 时，都必须按这 6 段编号回扣。

## C. 任务分类 A/B/C/D（来自 02 章）

| 类别 | 典型场景 | 最少需要几代杠杆 |
| --- | --- | --- |
| A | 一次性文本转换（翻译、摘要、改格式） | 仅 Prompt |
| B | 小范围代码改动（加个字段、写个简单接口） | Prompt + 少量 Context |
| C | 跨模块改动（UC-05 冻结读者、改响应体） | Prompt + 完整 Context |
| D | 持续性工作（回归测试、夜间生成、灰度分析） | 必须上 Harness |

## D. 模型能力维度（模型篇使用）

全书只从这 5 个维度评价模型：

1. **代码（Code）**：写 / 改 / 读 Java 等强类型代码。
2. **推理（Reason）**：多步决策、链式因果、算法题。
3. **写作（Write）**：中文长文、结构化文档。
4. **多模态（Multi）**：图文 / 代码截图 / 图表理解。
5. **长上下文（Long）**：≥ 100k token 的长文档稳定性。

> **禁止**在正文里用"智商 / 情商"之类的拟人词衡量模型。

## E. 成本 / 时间类术语

| 术语 | 定义 |
| --- | --- |
| Token | 模型计费与上下文长度的最小单位。本书在粗估时用 "1 汉字 ≈ 1.5~2 token"。 |
| 单次成本 | 一次调用（输入 + 输出）的美元 / 人民币估价。 |
| 单次耗时 | 从提交到拿到完整响应的墙钟时间（首 token 时间另记 TTFT）。 |
| 总成本 | 单次成本 × 重试次数 + 人工修返工工时 × 时薪。**本书所有"省 token"讨论都要回到这个口径**。 |
| 返工率 | 模型首版输出**未通过评测集**的比例。 |

## F. 评测 / 护栏术语

| 术语 | 定义 |
| --- | --- |
| 黄金用例集 | 一组**不变的**输入 + 期望输出，用于每次换模型 / 换 prompt 前先跑一遍。 |
| 护栏（Guardrail） | 在 harness 里硬编码的"不允许 AI 做的事"（改数据库 schema、删文件、调外部 API 等）。 |
| 红队测试 | 故意构造"诱导违规"的输入，看 AI 是否被带偏。 |

## G. RAG 相关

| 术语 | 定义 |
| --- | --- |
| RAG | Retrieval-Augmented Generation。**不是**"往 prompt 里塞一堆文本"，而是"在生成前按需检索 + 组织"。 |
| 切片（Chunk） | 把长文档切成 300~800 token 的段，保留 2~3 句重叠。 |
| 召回 / 精排 | 先大范围拿回候选，再用小模型或规则精排。 |
| 向量漂移 | 同一概念在不同文档的写法差异导致检索失败。 |

## H. 本书不使用或慎用的词

- **"颠覆" / "革命" / "Game Changer"**：一律禁用。
- **"智能涌现"**：慎用，仅在引用原论文时出现。
- **"人格化"描述**（模型"想" / "以为" / "故意"）：禁用，改为"模型输出……"。
- **"AGI"**：仅在 15 章结语讨论"下一代"时可出现一次。
- **"Agent" vs "智能体"**：正文统一使用英文 **`Agent`**；仅首次出现可括注"（智能体）"，之后不再使用中文译名。
- **"Prompt / Context / Harness" 的中文译名**：除首次括注外不再使用（遵循 `STYLE.md §7.1`）。
