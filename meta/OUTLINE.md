## 1. 章节明细与篇幅预算

> 列格式：**字数预算（汉字）| 一句话交付（草稿，写作时以最终章为准）| 核心示例（mini-library UC 锚点）| 关键「前后对照」**

### 00 前言（~2,000 字）
- 交付：你能判断这本书是不是写给你的，以及要按顺序读还是跳读。
- 示例：无代码；只有读者画像（3 种典型读者）。
- 对照：无。

### 01 地图：三代杠杆对照（~4,000 字）
- 交付：你能用"prompt / context / harness"三个名字，定位团队当前卡在哪一代。
- 示例：**同一个 UC-01（借书接口）** 在三代杠杆下分别怎么做（Prompt 一行、Context 包裹、Harness 流水线）。
- 对照 #1：三代并排对比表格（时间 / token / 返工率）。

### 02 能力边界：任务分类 A~D（~3,000 字）
- 交付：你能在接到新需求 60 秒内，把它归入 A/B/C/D 四类，决定用几代杠杆。
- 示例：把 mini-library 的 5 个 UC 逐个分类。
- 对照 #2：同一个"UC-05 冻结读者"被分到 B 和 C 的两种处理方式。

### 03 Prompt 篇（一）· 2022 气质与骨架模板（~3,500 字）
- 交付：你能把一条模糊需求改写成「角色 + 输入 + 约束 + 输出格式」4 段式 Prompt。
- 示例：给 UC-01 写 Prompt，一次生成 controller + service + mapper 骨架。
- 对照 #3：同一需求的 "2022 式祈使句" vs "2026 式结构化 prompt"。

### 04 Prompt 篇（二）· 反模式与提示资产化（~4,000 字）
- 交付：你能把散落在聊天记录里的 Prompt，收成带版本号、能 A/B 的资产。
- 示例：mini-library 的「代码生成 Prompt v1 → v7」演化史（附变更 diff）。
- 对照 #4：一次性 Prompt vs 资产化 Prompt 的维护成本。
- 对照 #5：**空洞夸赞 Prompt**（"请你成为世界上最厉害的 Java 大师"）vs **可验证 Prompt**。

### 05 Context 篇（一）· 上下文包裹顺序（~3,500 字）
- 交付：你能按固定顺序（系统规则 → 项目约束 → 任务描述 → 示例 → 代码上下文 → 输出要求）组织 payload。
- 示例：UC-05「冻结读者」的完整 context payload（XML + Markdown 混排）。
- 对照 #6：乱序 context vs 固定顺序 context 的输出稳定性。

### 06 Context 篇（二）· RAG 纪律（~3,500 字）
- 交付：你能判断"哪些内容该放进 RAG、哪些该放进 prompt、哪些根本不该喂给模型"。
- 示例：把 mini-library 的阿里规约、领域词表、历史 Bug 库分别归类，并设计检索粒度。
- 对照 #7：把整本规约塞进 prompt vs 切片 + 按需检索的 token 差异（附表）。

### 07 Context 篇（三）· 代码场景的上下文（~3,500 字）
- 交付：你能让 AI 读懂一个"跨 5 个文件"的改动，而不是只看到一个函数。
- 示例：UC-05 触发冻结 → 要读 `ReaderService` / `BorrowRecordMapper` / `RuleConfig` / `FreezeJob` / 测试类。
- 对照 #8：只贴被改函数 vs 贴"调用链 + 约束文件 + 相邻测试"的修复一次通过率。

### 08 Harness 篇（一）· 最小五要素（~3,500 字）
- 交付：你能在不引入任何框架的情况下，搭一个"角色 + 工具 + 记忆 + 评测 + 护栏"五件套的最小 harness。
- 示例：给 mini-library 搭一个"自动补全 Mapper + 单测"的 5 要素 harness（纯 shell + prompt 组合）。
- 对照：略（本章偏建模，对照放 10 章）。

### 09 Harness 篇（二）· 团队规范即缰绳（~4,000 字）
- 交付：你能把团队已有的 checkstyle、PR 模板、CI 规则，翻译成 AI 执行时的硬约束。
- 示例：把 `CASESPEC.md §5` 的 8 条硬约束，注入 harness 的系统提示 + pre-commit + review agent。
- 对照 #9：不注入规范的"漂亮但不合规"代码 vs 注入规范后的输出。

### 10 Harness 篇（三）· 评测与黄金用例集（~4,000 字）
- 交付：你能为 mini-library 搭一套 20 条黄金用例，每次换模型 / 换 prompt 先跑一遍再上线。
- 示例：UC-01 / UC-05 的黄金集（输入、期望、断言、阈值）。
- 对照 #10：无评测集"拍脑袋上线" vs 黄金集 gate 的回滚率对比。

### 11 模型篇（一）· 能力地图与成本时钟（~4,000 字）
- 交付：你能按"代码 / 推理 / 写作 / 多模态 / 长上下文"5 维，给截至 2026 年初的主流模型画能力雷达。
- 示例：在 mini-library 的 5 个 UC 下，分别列出"用 ____ 模型的效果 / 单次成本 / 单次耗时"。
- **点名到版本**：本章必须写出具体型号与发布时间，**不使用"某家旗舰"这种模糊措辞**。候选清单（写作时以 2026-04 可公开查证者为准）：
  - Anthropic：Claude Sonnet 4.5 / Claude Opus 4.x
  - OpenAI：GPT-5 / GPT-5 mini
  - Google：Gemini 2.5 Pro / Gemini 2.5 Flash
  - DeepSeek：DeepSeek V3.x / DeepSeek R1 系列
  - 阿里：Qwen3-Max / Qwen3-Coder
  - 字节：Doubao-1.5 Pro / Seed 系列
  - 月之暗面：Kimi K2
  - 智谱：GLM-4.6
- 对照：同一任务上"便宜档 vs 旗舰档"的效果差距与单价差（含表）。
- 版本注脚：每个版本名第一次出现，**加"（截至 2026-04）"小字注脚**，并在附录 B 指向可查证的官方博客。

### 12 模型篇（二）· 组合使用：plan / code / review / test（~4,000 字）
- 交付：你能为一个中型项目，设计"规划用旗舰、编码用次旗舰、review 用旗舰、单测用便宜档、批量回归用本地"的分工。
- 示例：UC-01 的完整迭代（需求→设计→编码→review→测试→回归）的分工链路，**每一步都标注具体模型版本号**。
- 对照：单模型 all-in-one vs 分工链路的总成本 + 总耗时。

### 13 Token 篇（一）· Token 花在哪 & 省钱矩阵（~3,500 字）
- 交付：你能拆出一次对话里 token 花在"系统提示 / 上下文 / 工具调用回包 / 历史 / 输出"的 5 个坑位， 以及大模型是如何通过Cache来帮你省钱的。
- 示例：对 mini-library 的一次"UC-01 修 bug"会话做 token 断面分析。
- 对照：优化前 vs 优化后（带数字：输入 tok / 输出 tok / $ / 秒）。

### 14 Token 篇（二）· 风险边界与总成本（~4,000 字）
- 交付：你能识别"省了 token 却更贵"的三种陷阱（返工、丢 context、漏检），并量化总成本。
- 示例：强行裁剪 context 导致 UC-05 漏判 → 线上 bug → 人工修复的总账。
- 对照：**省 token 返工 3 轮** vs **不省 token 一次过** 的总账对比表。


### 15 结语 · 三年后（~2,000 字）
- 交付：你能把这本书的"三代 + 2 章"框架，迁到下一代杠杆上（比如 2028 年的 XX engineering）。
- 示例：无新代码，只回扣前 14 章的"可迁移条目"。
- 对照：无。

## 2. 「前后对照」分布核查

共 10 组（#1 ~ #10），分布：

- 开篇区 2 组（01、02）
- Prompt 篇 3 组（03 × 1、04 × 2）
- Context 篇 3 组（05 × 1、06 × 1、07 × 1）
- Harness 篇 1 组（09）
- 评测篇 1 组（10）

模型篇、Token 篇另有表格形式的对照，不计入这 10 组，避免冲抵。

## 3. 附录规划（不占正文预算）

- **附录 B 延伸阅读**：去工具化的阅读路径（论文 / 博客文章类型，不锁具体链接版本）。
- **附录 C 术语扩展**：`GLOSSARY.md` 的展开版，每个术语 300~500 字的长解释。

## 5. 篇间过渡（在 `scripts/build_book.py` 合并时插入）

- 开篇 → Prompt 篇：《为什么 Prompt 仍然是每一代的底座》
- Prompt 篇 → Context 篇：《Prompt 的天花板在哪里》
- Context 篇 → Harness 篇：《当 context 也不够用》
- Harness 篇 → 模型篇：《选对模型，比加一层 harness 还省事》
- 模型篇 → Token 篇：《每一次"选"都是在花钱买时间》
- Token 篇 → 结语：《便宜、快、对：永远只能选两个？》

每段过渡约 150~250 字，不占章节预算。

## 6. 写作接力顺序建议

优先写**完成度需求最强**的两章作为锚：

1. 先写 **01 地图** + **CASESPEC 内的 GOOD/BAD 锚**（决定全书口径）。
2. 再写 **09 Harness 团队规范即缰绳**（把硬约束写死）。
3. 然后按 `00 → 02 → 03 → 04 → 05 → 06 → 07 → 08 → 10 → 11 → 12 → 13 → 14 → 15` 顺序铺。
4. 最后补附录。

> 每完成 3 章，更新一次 `meta/CONTINUITY.md`（尚未创建，留到开写时再建）。
