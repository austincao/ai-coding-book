# AGENTS.md — 《AI Coding 工程化》协作与编辑约束

> 本文供 **人类作者** 与 **AI 助手** 在续写、修订、建读本仓库时阅读。  
> **体例与排版**以 `meta/STYLE.md` 为准；**案例事实**以 `meta/CASESPEC.md` 为唯一真理源；**本文**沉淀协作中形成的偏好与易错点。

---

## 1. 仓库结构与改稿原则

| 路径 | 作用 |
| --- | --- |
| `chapters/*.md` | **正文源稿**：一切实质性修改优先在这里完成。 |
| `appendix/*.md` | 附录源稿。 |
| `meta/STYLE.md` / `meta/OUTLINE.md` / `meta/CASESPEC.md` | 体例、提纲、贯穿案例规格。 |
| `scripts/build_book.py` | 合并章节、注入封面与阅读导航，输出 `dist/`。 |
| `dist/全书.md` / `.html` / `.pdf` / `.epub` | **构建产物**：勿手改；改源稿后重新构建。 |

**构建命令**（源稿变更后应执行以更新交付物）：

```bash
python scripts/build_book.py
```

**改稿粒度**：只改当前任务需要的文件；避免无关“顺手重构”、避免未征得同意的大范围重命名全书术语（除非有单独迁移任务）。

## 2. 作者偏好（叙事与立场）

- **读者称呼**：全书用「你」；不神化模型；承认工具会变，强调**可迁移的工作方式**。
- **读者分层**：可用「以写代码为主 / 技术负责人 / 产品业务方」等**职责描述**，**避免 A/B/C 字母标签**给读者贴代号；若必须分档，说明与第二章任务档 **A/B/C/D** 不是同一套字母。
- **前言编号**：前言内小节用「一、二、三…」，不用「0.1、0.2」（易与正文章节编号混淆）。
- **叙事入口**：允许「咨询/我见过」类开篇；**不强行**把每章都换成同一种开头（除非作者要求统一变体）。
- **结语收束**：偏平实、少口号化排比；避免空洞“祝你比我强”式堆砌（已按此方向收过一轮）。

## 3. 术语与译法（易争议处）

以下为协作中明确过或正在向之对齐的方向，**批量替换前应与作者确认**。

| 主题 | 约定 |
| --- | --- |
| **三代主线** | 书名级概念长期写作「三代杠杆」；若读者反馈「杠杆」费解，正文可并列 **「三代发力点 / 着力点 / 能力层级」** 等白话，首见时一句定义即可，不必全书机械替换。 |
| **Harness 比喻** | **「基座/工作台」**优先于「工位」（「工位」易联想到办公室格子间）。英文 **Harness** 可保留；括号内可用「框架工程 / 工作台工程」与第一章现文一致。 |
| **任务分档** | 第二章 **A/B/C/D** 表示任务档位；与读者画像字母区分见上。 |
| **贯穿案例** | 统一 `mini-library`；规则、数值、类名以 `meta/CASESPEC.md` 为准，禁止悄悄改设定。 |
| **用例编号 vs 正文** | `CASESPEC`、评测目录与 JSON `id` 使用 **UC-01～UC-05**；**正文叙述以业务名**（借书、还书、读者冻结等）为主，需对齐规格或文件时写 **（UC-xx）**。前言 §五 有对照表；勿改代码块里与仓库一致的 `UC-01-happy` 等机器名。 |

## 4. 数字、账单与示例一致性（高频事故）

- **Claude Opus 4.7 API 单价**：全书示例、对照表以 **`chapters/13-token-花在哪与省钱矩阵.md`**（第十三章）中的官方单价表为**准**；第十一章等若有涉及 Opus，须与第十三章一致。
- **美元与人力折算**：书中用于**国内读者示意**的工程师时薪曾约定为 **$25**，与 **$200/日** 等业务假设在同一场景内不自相矛盾；改动时薪需**通读**同章表格与小节结论。
- **心算检查**：「坏做法 vs 好做法」倍数（例如约 12～13 倍）应与表中分子分母同批更新。
- **日期**：非必要不出现具体某日；可用「本书定稿前后」「填当月」等模糊占位，避免虚构精确日程与事实冲突。

---

## 5. 结构与前后不打架

- **章节指称**：同一本书内尽量统一「第 N 章」或「第三章、第四章」一种风格；避免「03 / 第四章」混排。
- **顺叙**：后文不得写「上一章已经讲过」而实际章节在读者顺序上在后（若需指 Harness 缰绳，用「下一章」或「后文第九章」）。
- **两套编号**：第五章 **六段 Context** 的「⑤ Code」与第七章 **代码五圈** 的「⑤ 规范层」易混；**Code 段示例块内**应有一句说明，避免读者把两套数字读成一个体系。

---

## 6. 文体：减少「AI 味」与无效重复

- **少用**：空洞过渡（「综上所述…」）、万能套话、无信息增量的形容堆砌。
- **检查**：同一章内是否**重复同一结论**两处仅换说法（可合并一段）。
- **第三章 vs 第九章**：「否定句 / 禁止清单」——第三章反对的是**长串裸「不要」**；第九章的**带编号反例 + 单条禁止**与之兼容；若写「禁止有效」，应点明与第三章区分，避免读者认为矛盾。

---

## 7. 校对与外来规范

- 若作者提到「按微信读书 / Editor Agent 标准」校对：可参考仓库内 **`Editor Agent.md`** 的维度（冗余、合规、可读性、技术准确性）；**具体修改**仍以本书 `STYLE.md` 与作者口头约束为准。

- [ ] 改的是 `chapters/` 或 `appendix/`，而非仅 `dist/`。

- [ ] 无未说明的敏感真实数据；示例 ID 仍为 seed 约定（如 `bookId=1001`）。

## 9. 文档边界

- **不写**：除非作者要求，不新建大量与成书无关的说明文档；本文与 `STYLE.md` 有分工，避免复制粘贴全书模板。
- **可更新**：协作中出现新的全书级约定时，**追加本节或新开一小节**，并注明大致日期或版本（可选）。
